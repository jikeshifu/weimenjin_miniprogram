# 在线更新策略

微门禁开源版本采用“完整后台主版本包 + 基线清单 + 数据库结构修复”的更新策略，避免增量包遗漏模板、静态资源、接口控制器或数据库脚本。

## 发布策略

1. 每次发布先确定最新代码基线，生成完整后台更新包。
2. 发布脚本自动生成 `baseline_版本号.json`，记录每个文件的路径、大小和 SHA256。
3. `manifest.json` 记录版本号、更新包地址、更新包 SHA256、基线清单地址、文件数量、包大小和数据库脚本列表。
4. 更新包包含后台运行所需代码和静态资源，排除用户运行数据。
5. 数据库变更统一放入 `database/update_*.sql` 或 `database/updates/*.sql`，发布时自动写入清单。

## 升级策略

1. 有明确版本号时，后台按版本号判断是否需要升级。
2. 没有版本基线或版本号不可靠时，按完整后台主版本包对齐代码。
3. 升级前自动备份代码和数据库。
4. 升级前读取并单独备份旧 `config/my.php`。
5. 升级时保留 `.env`、`env`、数据库连接配置、运行配置、上传目录、二维码目录、证书目录、运行缓存和历史更新包。
6. 升级后自动执行数据库脚本、检测关键表字段、补齐应用配置。
7. 如果旧版本的业务配置只存在于 `config/my.php`，系统会在数据库表创建或修复后，将可识别配置迁移到 `cd_appconfig`；已在数据库中填写过的非空配置不会被覆盖。
8. 配置迁移完成后，再从数据库重新生成新的 `config/my.php`。
9. 更新完成后清理缓存，后台页面显示版本、备份路径、数据库脚本执行结果、配置迁移结果和更新日志。

## 发布校验

发布前必须检查：

- 更新包 SHA256 与 manifest 一致。
- 更新包包含登录页背景、camweb、应用配置、摄像头二维码、硬件云和数据库修复相关文件。
- 更新包不包含用户私有配置、上传文件、证书、二维码、运行缓存和旧更新包。
- 旧版本只有 `config/my.php` 时，升级流程能备份并迁移业务配置到数据库。
- 编码检查通过，中文说明保持 UTF-8。
- 在演示平台按后台“系统更新”流程升级一次，不直接手工覆盖文件作为最终验证。
